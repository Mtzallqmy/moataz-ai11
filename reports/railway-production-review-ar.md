# تقرير مراجعة وإصلاح Moataz AI للنشر على Railway

## النتيجة التنفيذية

كان المشروع يفشل في مرحلة البناء قبل تشغيل الخادم. بعد الإصلاح أصبحت أوامر ESLint وTypeScript وبناء الواجهة والخادم ناجحة، وأصبح إعداد Railway يفشل بوضوح عند نقص قاعدة البيانات أو المتغيرات بدل إظهار نشر Active غير صالح.

## الأسباب المؤكدة للفشل

1. وجود مجلد `server/src/database` يستورد `drizzle-orm` وملف `drizzle.config.ts` يستورد `drizzle-kit`، بينما الحزمتان غير موجودتين في `package.json`.
2. طبقة Drizzle غير المستخدمة تشير إلى خصائص إعداد غير موجودة مثل `databasePoolMax` و`databaseSslCa`، وتستخدم `query_timeout` بطريقة لا تقبلها أنواع `pg` الحالية.
3. `server/tsconfig.json` يضم جميع ملفات `server/src/**/*.ts`؛ لذلك الكود غير المستخدم كان يكسر `npm run build:server` على Railway.
4. `better-sqlite3` كان اعتماد تشغيل أساسيًا native، مع أن الإنتاج يعتمد PostgreSQL؛ أي فشل لتنزيل binary أو تجميع C++ كان يوقف `npm ci` كاملًا.
5. Railway كان يفحص `/api/health` الذي يعيد 200 حتى في وضع `configuration_required`. النتيجة: Railway قد يعرض الخدمة Active بينما التطبيق غير جاهز ويرجع 503 للمستخدم.
6. اختبار المزودات كشف أن Base URL بدون scheme أو رابط root لمزود معروف لا يطبّع كما هو متوقع.
7. المستودع احتوى Workflows وملفات bootstrap/audit مؤقتة مخصصة لفرع تطوير، وليست جزءًا من مسار إنتاج مستقر.

## الإصلاحات المنفذة

- إزالة طبقة Drizzle الجزئية وغير المستخدمة وملفاتها، مع إبقاء طبقة قاعدة البيانات الفعلية التي تستعملها جميع المسارات.
- نقل `better-sqlite3` إلى `devDependencies` وتحميله ديناميكيًا عند اختيار SQLite فقط، مع حذفه من صورة تشغيل Railway.
- منع SQLite في `production` وRailway؛ PostgreSQL أصبح شرط جاهزية واضحًا.
- جعل Docker يتجاهل الاعتماد الاختياري SQLite، وإزالة أدوات C/C++ غير المطلوبة من Build image.
- إضافة إعدادات pool وconnection/idle/statement timeouts وretry لقاعدة البيانات.
- دعم `DATABASE_SSL_CA` عند `verify-full`، مع تحقق مبكر من الإعداد.
- جعل الترحيلات قابلة للتكرار بصورة أكثر أمانًا، ومعالجة تعارض unique عند تزامن تشغيل نسختين.
- تحديث `railway.json` إلى schema الحالي وتعيين `/api/ready` كـhealthcheck بمهلة 180 ثانية.
- تحديث Docker HEALTHCHECK إلى `/api/ready`.
- إصلاح تطبيع روابط المزودات وإضافة `https://` تلقائيًا للـhostname، وإرجاع المسار الافتراضي عند إدخال root لمزود معروف.
- حذف Workflows المؤقتة وملفات audit/bootstrap، والإبقاء على CI الرئيسي فقط.
- تحديث التوثيق ومتغيرات النشر للإصدار 1.5.1.

## التحقق الذي تم تنفيذه

- `npm run lint`: ناجح.
- `npm run typecheck`: ناجح.
- `npm run build`: ناجح للواجهة والخادم.
- 39 اختبار وحدة غير معتمد على قاعدة بيانات native: ناجحة.
- 23 عبارة migration لSQLite تم تنفيذها في قاعدة مؤقتة والتحقق من صحتها.
- تشغيل النسخة المبنية في وضع نقص الإعدادات: `/api/health` أعاد 200 و`/api/ready` أعاد 503 مع أسماء المتغيرات الناقصة دون أسرار.

## قيد بيئة المراجعة

بيئة الفحص الحالية تعمل بـNode.js 22 بينما المشروع يستهدف Node.js 20، ولم يتوفر binary محلي لـ`better-sqlite3` بسبب تعذر تنزيله وتجميع SQLite C ضمن المهلة. لذلك لم تُشغّل اختبارات التكامل المعتمدة على SQLite هنا. مسار Railway الإنتاجي لا يثبت SQLite أصلًا، وCI الموجود يستخدم Node.js 20 لتشغيل المجموعة الكاملة.

## متطلبات ما قبل النشر

- PostgreSQL/Supabase URL صالح.
- `JWT_SECRET` و`ENCRYPTION_KEY` مختلفان وطول كل منهما 32 محرفًا على الأقل.
- دومين Railway مولّد ومضاف إلى `APP_URL` و`CORS_ORIGIN`.
- كلمة مرور المشرف 12 محرفًا على الأقل.
- إبقاء `ALLOW_SHELL=false` و`TELEGRAM_POLLING=false` حتى إعداد الخدمات المطلوبة.
