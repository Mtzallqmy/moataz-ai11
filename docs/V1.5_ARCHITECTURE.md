# Moataz AI 1.5 — Production Architecture

## Provider reliability

OpenAI-compatible providers use a validated direct HTTP transport for `/models` and `/chat/completions`. Base URLs are normalized before storage and request construction. Provider authentication failures are upstream configuration errors and never invalidate the Moataz AI application session.

Provider verification discovers model IDs when available, probes bounded candidates with real completion requests, records every attempt, selects the first working model, and updates conversations that use the provider. Values such as `auto`, `default`, or `Free` are discovery instructions rather than model IDs.

## Sessions

Access tokens are short-lived. Rotating refresh tokens are stored as hashes in PostgreSQL and sent through HttpOnly cookies. Sessions survive page reloads and transient API/provider failures. Users explicitly confirm logout and can revoke other devices from Settings.

## Attachments

Chat uploads are stored under the authenticated user's isolated workspace. Filenames are normalized, writes are atomic, hashes are recorded, and attachment ownership is checked against both user and chat. Text/code is bounded before model context injection, supported images are passed to multimodal providers, ZIP files are indexed without executing archive contents, and binary files remain available for sandbox processing.

A failed message request unbinds its attachments and removes the provisional user message so the same client draft can be retried safely.

For persistent Railway uploads, mount a volume at the configured `WORKSPACE_DIR` such as `/app/workspace`.

## Multi-agent mode

Multi-agent conversations require at least two verified providers. Up to three providers receive bounded independent roles, and the primary provider synthesizes successful reports. Individual failures are recorded without hiding successful agent output.

## API console and sandbox

The API console invokes public HTTP APIs through the confirmed `http_request` tool. It applies SSRF protection, response-size limits, timeouts, header filtering, and structured upstream errors.

Shell commands never execute inside the main Railway application container. Deploy `sandbox-service/` as a second service with its own `SANDBOX_TOKEN`, no unrelated secrets, an optional dedicated volume, and platform resource limits. The contract is `GET /health` and authenticated `POST /v1/execute`.
