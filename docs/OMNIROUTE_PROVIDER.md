# دمج OmniRoute كمزوّد في Moataz AI

## القرار المعماري

يُستخدم OmniRoute كبوابة مستقلة متوافقة مع OpenAI، وليس كحزمة مضمّنة داخل عملية Moataz AI. السبب أن OmniRoute الحالي يتطلب Node.js 22 أو 24+، بينما مشروع Moataz AI يستهدف Node.js 20 على Railway. فصل الخدمتين يحافظ على استقرار البناء، ويتيح تحديث OmniRoute أو تعطيله دون كسر التطبيق الأساسي.

لم يُنسخ كود OmniRoute داخل هذا المشروع. التكامل يستهلك واجهته العامة فقط. OmniRoute مرخّص بترخيص MIT؛ عند توزيع نسخة مشتقة منه يجب الاحتفاظ بإشعار حقوق النشر والترخيص الخاص به.

## التشغيل المحلي

1. ثبّت وشغّل OmniRoute في جهاز أو خادم يدعم Node.js المطلوب:

```bash
npm install -g omniroute
omniroute
```

2. افتح لوحة OmniRoute على:

```text
http://localhost:20128
```

3. من **Providers** اربط المزودات أو الحسابات التي تملك حق استخدامها.
4. من **Endpoints** أنشئ API key خاصًا بالبوابة. هذا هو المفتاح الذي يضاف إلى Moataz AI، وليس بالضرورة مفاتيح المزودات الأصلية.
5. داخل Moataz AI افتح إعدادات المزودات وأدخل:

```text
Name: OmniRoute
Type: OmniRoute Gateway
Protocol: OpenAI-compatible
Base URL: http://127.0.0.1:20128/v1
API Key: مفتاح Endpoints الذي أنشأه OmniRoute
Model: auto
Streaming: enabled
```

6. اضغط **اكتشاف النماذج**. ستظهر Model IDs الحية التي يعيدها `/v1/models`، إضافة إلى نماذج OmniRoute الافتراضية الآتية:

```text
auto
auto/coding
auto/fast
auto/cheap
auto/offline
auto/smart
```

يمكن أيضًا استخدام Model ID صريح مثل `provider/model` إذا كان ظاهرًا في كتالوج OmniRoute.

## التشغيل على Railway أو خادم بعيد

شغّل OmniRoute في **خدمة مستقلة** تدعم Node.js المطلوب، ثم عرّضها عبر HTTPS. لا تستخدم `127.0.0.1` أو `localhost` بين خدمتين منفصلتين؛ كل خدمة لها شبكة/حاوية خاصة.

مثال إعداد Moataz AI:

```text
Base URL: https://omniroute.example.com/v1
API Key: Endpoint key generated by OmniRoute
Model: auto/coding
```

لا تضف OmniRoute إلى `package.json` الخاص بـMoataz AI ولا تشغله داخل نفس Dockerfile. هذا يمنع تعارض إصدار Node واعتماديات SQLite/Next.js الكبيرة مع خادم Express الحالي.

## مسار الطلب

```text
Moataz AI UI
  → Moataz AI backend
  → Authorization: Bearer <OmniRoute endpoint key>
  → OmniRoute /v1/chat/completions
  → routing/combo/fallback configured in OmniRoute
  → selected upstream provider/model
```

Moataz AI لا يرى مفاتيح المزودات المخزنة داخل OmniRoute، وOmniRoute لا يحصل على مفاتيح المستخدمين المخزنة لمزودات Moataz AI الأخرى.

## Custom Headers الاختيارية

يسمح تكامل OmniRoute بهذه الرؤوس فقط إلى جانب القائمة الآمنة العامة:

```text
X-OmniRoute-Mode
X-OmniRoute-Budget
X-OmniRoute-Compression
X-Route-Model
```

لا يمكن للـCustom Headers استبدال `Authorization` أو `Host` أو رؤوس النقل الحساسة.

## التحقق والتشخيص

اختبر البوابة مباشرة من الخادم الذي يشغّل Moataz AI:

```bash
curl -fsS https://omniroute.example.com/v1/models \
  -H "Authorization: Bearer $OMNIROUTE_ENDPOINT_KEY"
```

ثم اختبر من واجهة Moataz AI عبر **اكتشاف النماذج** و**اختبار الاستدلال**. لا تعتبر `auto` وصولًا سحريًا إلى أي نموذج: OmniRoute يستطيع توجيه الطلب فقط إلى المزودات والحسابات والنماذج التي ربطتها أنت والتي تسمح خططها وشروطها بالاستخدام.

## أعطال شائعة

- **401**: مفتاح Endpoint غير صحيح أو لم يُرسل.
- **403**: Endpoint أو النموذج أو الحساب لا يملك الصلاحية المطلوبة.
- **404 على `/models`**: راجع أن Base URL ينتهي بـ`/v1` ولا يحتوي `/models` مرتين.
- **Connection refused محليًا**: OmniRoute غير مشغّل أو المنفذ ليس `20128`.
- **Railway لا يصل إلى localhost**: استخدم HTTPS العام لخدمة OmniRoute المنفصلة.
- **`auto` مرفوض مع مزود آخر**: هذا مقصود؛ نماذج `auto/*` مسموحة فقط لمزوّد OmniRoute.
