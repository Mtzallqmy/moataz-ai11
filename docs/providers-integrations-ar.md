# المزوّدات والتكاملات — سلوك الإنتاج

## المزوّدات

- حفظ إعداد المزوّد لا ينفذ طلبًا مدفوعًا إلى الجهة الخارجية.
- الحالة بعد الحفظ هي `untested`.
- اختبار الاتصال منفصل، ويحدّث الحالة إلى `verified` أو `failed`.
- أخطاء المصادقة، الرصيد، النموذج، حدود الطلبات، الشبكة والمهلة تعاد برمز HTTP مناسب وتفاصيل منقحة من الأسرار.
- حذف المزوّد يفصله عن المحادثات المرتبطة به؛ عند الإرسال يطلب النظام اختيار مزوّد صالح بدل إرجاع خطأ داخلي.

## GitHub وTelegram

- تُشفّر التوكنات قبل التخزين.
- الحفظ منفصل عن الاختبار، لذلك يمكن تعديل الإعداد ثم اختباره من الواجهة.
- لا تُستخدم التكاملات داخل الأدوات أو Telegram polling إلا بعد نجاح الاختبار ووصول الحالة إلى `verified`.
- يدعم Telegram قائمة `allowedChatIds` لمنع استقبال الرسائل من محادثات غير مصرح بها.
- عند تفعيل `TELEGRAM_POLLING=true` يعيد الخادم تحميل البوتات بعد نجاح الاختبار أو تعديل التكامل أو حذفه، دون الحاجة إلى إعادة نشر يدوية.

## حالات التحقق

| الحالة | المعنى |
|---|---|
| `untested` | تم الحفظ ولم يُختبر الاتصال بعد |
| `verified` | نجح اتصال فعلي بالجهة الخارجية |
| `failed` | فشل آخر اختبار وتوجد شفرة خطأ تشخيصية |

## أخطاء شائعة

- `provider_authentication`: مفتاح المزوّد غير صحيح.
- `provider_billing`: الرصيد أو الفوترة لا تسمح بالطلب.
- `provider_model_not_found`: النموذج غير موجود أو غير متاح للحساب.
- `provider_rate_limit`: تجاوز حدود الطلبات.
- `integration_authentication`: توكن GitHub أو Telegram غير صحيح أو ملغى.
- `telegram_token_invalid_format`: التوكن لا يطابق صيغة BotFather.
- `provider_required`: يجب تعيين مزوّد للمحادثة.

تظهر الواجهة رسالة الخدمة الخارجية المنقحة ورقم الطلب، ولا تعرض التوكن أو مفتاح API أو بيانات اتصال قاعدة البيانات.
